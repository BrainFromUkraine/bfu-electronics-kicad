# BFU Electronics — правила роботи для агента

Це бібліотека футпринтів KiCad для проєктів **Brain From Ukraine**.
Головний споживач — проєкт **SICH** (LoRa-комунікатор на ESP32-S3).

## Головне правило: НЕ ВИДУМУЙ

Жоден розмір, координата, крок падів чи номер піна **не беруться з пам'яті**.
Кожне число має походити з одного з перевірених джерел:

1. **Офіційний даташит виробника** — розділ *Recommended PCB Land Pattern* /
   *Mechanical Drawing*. Найвищий пріоритет.
2. **Офіційна бібліотека виробника** (напр. `espressif/kicad-libraries`).
3. **Офіційна бібліотека KiCad** (`kicad-footprints`).
4. **LCSC / JLCPCB / SnapMagic** — лише якщо перших трьох немає,
   і обов'язково звірити з кресленням даташита.

Якщо джерела немає — **компонент не додається**. Краще порожня категорія,
ніж футпринт, через який доведеться перевипускати плату.

У полі `descr` кожного футпринта вказуй джерело розмірів.

## Структура репозиторію

```
BFU_Electronics/          <- сама бібліотека (те, що підключається в KiCad)
├── BFU_RF.pretty/        LoRa, SX126x, радіомодулі
├── BFU_MCU.pretty/       ESP32-S3, WROOM
├── BFU_Display.pretty/   TFT (ST7789), OLED, e-Paper
├── BFU_Audio.pretty/     INMP441, MAX98357A
├── BFU_GNSS.pretty/      GPS/GNSS
├── BFU_Power.pretty/     LDO, DC-DC, зарядка Li-Ion
├── BFU_Connector.pretty/ USB-C, гребінки, JST, IPEX
├── BFU_Sensor.pretty/    IMU, датчики
└── fp-lib-table
scripts/validate.py       <- перевірка футпринтів перед комітом
docs/                     <- інструкції
datasheets/               <- PDF даташитів (джерела розмірів)
```

Тека бібліотеки **обов'язково** має суфікс `.pretty` — інакше KiCad її не бачить.
Один `.kicad_mod` = один компонент.

## Формат футпринта

- Версія: `(footprint "NAME" (version 20221018) (generator "BFU_Electronics")`
- Ім'я файлу = **точний партномер** виробника: `E22-900M30S.kicad_mod`
- Обов'язкові елементи:
  - `descr` — виробник, ключові параметри, габарити, тип корпусу, джерело
  - `tags` — для пошуку
  - `attr smd` або `attr through_hole`
  - `fp_text reference "U**"` на `F.SilkS`
  - `fp_text value "<партномер>"` на `F.Fab`
  - контур корпусу на `F.SilkS`
  - `F.CrtYd` — courtyard з відступом ~0.25–0.5 мм від габариту
  - маркер піна 1 (кружечок або зріз кута) на `F.SilkS`
- Для RF-модулів з PCB-антеною — позначити **зону keepout антени** на `F.Fab`
  штриховою рамкою + підписом.

## Робочий процес із новими файлами

Нові `.kicad_mod` з чату **зберігаються в теку `incoming/`, а не напряму
в `.pretty`**. Причина: браузер може мовчки перезаписати наявний файл, якщо
ім'я збігається — так уже двічі постраждав `MAX98357A.kicad_mod`.

Порядок:
1. файл кладеться в `incoming/`;
2. агент перевіряє, що внутрішнє ім'я `(footprint "...")` збігається з іменем
   файлу і що файл із таким іменем ще не існує в цільовій `.pretty`;
3. агент переносить файл у потрібну категорію;
4. `python scripts/validate.py`;
5. коміт.

Тека `incoming/` має лишатися **порожньою між сесіями** (крім `README.md`).

**Перед кожним комітом** перевіряй `git status` на предмет ` D` або ` M`
навпроти вже закомічених `.kicad_mod` — це ознака, що файл затерли або
видалили. Відновлюй такий файл з HEAD (`git restore <файл>`), а не комітуй
пошкодження.

## Перед комітом

```bash
python scripts/validate.py
```

Скрипт перевіряє: суфікс `.pretty`, наявність `descr`/`tags`, courtyard,
дублікати імен падів, порожні бібліотеки.

## Стиль комітів

```
add(BFU_RF): E22-900M30S — SX1262 1W LoRa module
fix(BFU_MCU): ESP32-S3-WROOM-1 courtyard 0.25mm clearance
docs: інструкція підключення для KiCad 9
```

## Контекст проєкту SICH

- MCU: **ESP32-S3-WROOM-1 N16R8** (октальний PSRAM → GPIO 35/36/37 зайняті)
- Радіо: **два модулі Ebyte E22** (потрібна RF keep-out зона між ними,
  друга гармоніка 433 МГц потрапляє в 900 МГц діапазон)
- Дисплей: **ST7789V** 240×320
- Клавіатура: **CardKB** (I2C 0x5F)
- Аудіо: **INMP441** (мікрофон, I2S), **MAX98357A** (підсилювач, I2S)
- GPS: модель уточнюється
- Плата: 88 × 122 мм, двошарова (B.Cu — компоненти й сигнали, F.Cu — GND-полігон)
